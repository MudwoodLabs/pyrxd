## CONFIRMED Findings

### CRITICAL

#### C1 — `(F-001)` Non-minimal value-param push bricks the covenant for asset values 1..16 (permanent fund-stranding)

- **Mechanism.** The covenant constructor substitutes the asset value via `_push(_scriptnum(amount))` at `htlc_covenant.py:313` (FT `amount`), `:360` (NFT `nftCarrierValue`), and `:403` (RXD `amount`). For a value in 1..16 this emits a 1-byte *direct* push (e.g. `amount=5` → bytes `0105`) instead of the required `OP_1..OP_16`. The push sits inside the *funded* covenant SPK in an always-executed position (FT/NFT: in the prologue before `OP_IF`; RXD: duplicated in both the claim `OP_IF` and refund `OP_ELSE` branches), so it executes on **both** the claim and refund paths. The contrasting `refundCsv` field at `:312/:359/:402` correctly uses `_minimal_num_push` — confirming the team knew the hazard and applied the safe encoder to one field but not the value param.
- **Evidence.** Verified by source read this round (the asymmetry at `htlc_covenant.py:312–313 / 359–360 / 402–403` is reproduced above). **PROVEN** on the isolated `radiant-core:v2.3.0` regtest node: a covenant built with `amount=5` produced `...8800cc0105a26900cdaa...`; `testmempoolaccept` of the production `build_htlc_claim_tx` spend returned `allowed=False, reject-reason="16: mandatory-script-verify-flag-failed (Data push larger than necessary)"`, while the `amount=1000` control returned `allowed=True`. The refund spend (after CSV maturity) returned the same mandatory failure. Consensus chain confirmed in source: `CheckMinimalPush` returns false for a 1-byte value 1..16 (`script.cpp:381-383`); `SCRIPT_VERIFY_MINIMALDATA` is in `MANDATORY_SCRIPT_VERIFY_FLAGS` (`policy.h:123-133`); enforced on every executed push (`interpreter.cpp:312-316`); a mandatory-flag failure is block-invalid via `CheckInputs`→`state.DoS(100,…,"mandatory-script-verify-flag-failed")` (`validation.cpp:1399-1425`), and `ConnectBlock` calls `CheckInputs` with consensus flags (`validation.cpp:1767`).
- **Exploitability / preconditions.** Triggered by any swap whose `radiant_amount` ∈ {1..16}. No guard stops it: `NegotiatedTerms.__post_init__` checks only `_pos_int` (`swap_state.py:246`); the `_guard_bd`/`_guard_refs`/`_opcode_bd_positions` walks (`htlc_covenant.py:178/219/228`) never check push minimality; `post_asset_lock_revalidate` compares observed vs expected SPK but both come from the same bricked builder, so they match and the FSM advances to BOTH_LOCKED. **Reachability is ordinary, not contrived:** for NFT, `nftCarrierValue` is the carrier sats — singletons routinely carry dust (1 sat), making NFT the most reachable; for FT, `amount` is the token quantity, and trading 1..16 units of a fungible token is normal. The mainnet dust run used `amount=1000` (multi-byte, minimal), which is the only reason it never tripped.
- **Verifier verdicts.** consensus-source: **CONFIRMED** (every load-bearing claim correct against ground truth). exploit-construct: **CONFIRMED** — reproduced on live regtest, both claim and refund bricked. existing-guard: **CONFIRMED** — no guard prevents it; the only minimality test (`test_htlc_covenant.py:121`) covers `refundCsv` exclusively.
- **Impact.** Permanent, unconditional, irrecoverable loss of the locked Radiant asset whenever the asset value is 1..16. It is a *guaranteed* brick (not probabilistic), bricks silently at lock time (revalidation uses the same builder), and strands the asset on **both** branches — so even mutual refund is impossible. In the asymmetric instantiation, the maker claims BTC (revealing `p`) while the taker's asset is stranded, but loss occurs regardless of intent.
- **Fix.** Replace `_push(_scriptnum(amount))` with `_minimal_num_push(amount)` at `htlc_covenant.py:313/:360/:403` (and the spike builder). Add a build-time guard running a `CheckMinimalPush`-equivalent over **every** push in the assembled `funded_spk` (not just the bare-`0xbd` walk), fail-closed on any non-minimal push. Extend `validate_htlc_covenant.py` to assert minimal encoding of the value operand for {1, 16, 17}.

### HIGH

#### H1 — `(F-002)` Radiant leg uses raw `t_rxd.value` as both the CSV operand and the spend `nSequence`, ignoring `TimeUnit`; a SECONDS-tagged `t_rxd` desyncs the on-chain refund window from every safety gate (counterparty theft)

- **Mechanism.** `RadiantCovenantLeg._build_covenant` passes `terms.t_rxd.value` *raw* as both the covenant `refundCsv` operand and the refund-spend `nSequence` (`radiant_leg.py:335/347/357`, `htlc_spend.py:272`), never calling `Timelock.to_nsequence()` and never checking the unit. The off-chain gates instead unit-normalize `t_rxd` to blocks via `policy.block_interval_s` (`swap_coordinator.py:366/501`). The same-unit ordering guard (`swap_state.py:265`) is skipped when `t_btc` is BLOCKS and `t_rxd` is SECONDS. The BTC leg, by contrast, correctly BIP68-encodes via `taproot.py:248-253` — so the two legs interpret the identical unit-tagged Timelock differently.
- **Evidence.** **PROVEN** by executing the real gates: `t_rxd = Timelock(4194404, SECONDS)` (= `(1<<22) | 100`) constructs fine; `assert_timelock_margin` PASSES (normalizes to `4194404//600 = 6990` blocks); `assess_claim_finality` returns SAFE believing the refund opens ~6990 blocks out. Consensus reads the raw value as a SECONDS lock: bit-22 `SEQUENCE_LOCKTIME_TYPE_FLAG` set → effective lock `(4194404 & 0xFFFF) << 9 = 100*512 = 51200s ≈ 14.2h` (`tx_verify.cpp:104-130`, `transaction.h:126-149`; `OP_CSV` masks operand and `nSequence` identically at `interpreter.cpp:2808-2835`). **CONFIRMED-BY-SOURCE:** no BLOCKS-only guard and no `0xFFFF` cap anywhere on the Radiant build path (grep).
- **Exploitability.** A malicious maker proposes `t_rxd = Timelock(4194404, SECONDS)`, `t_btc = Timelock(7000, BLOCKS)`. The taker's `pre_btc_lock_check` passes (REF, H-freshness, margin, post-lock revalidate all clear), and the taker funds BTC believing ~24 days of Radiant safety. The maker reveals `p` by claiming BTC, then CSV-refunds the asset at ~14h while the taker's coordinator still reports SAFE/WAIT — **maker keeps the asset AND takes the BTC.** The preconditions are entirely attacker-chosen terms; this is exactly the threat model the gates exist to defeat.
- **Verifier verdicts.** consensus-source, exploit-construct, existing-guard: all **CONFIRMED**. Latent today only because the dust harness hardcodes BLOCKS (`dust_swap_run.py:116`); the production `SwapCoordinator` + `RadiantCovenantLeg` + `NegotiatedTerms` accept arbitrary SECONDS terms.
- **Fix.** Fail-closed reject any non-BLOCKS `t_rxd` (and `refund_csv > 0xFFFF`) in `RadiantCovenantLeg._build_covenant` / `build_htlc_covenant_*`. Enforce BLOCKS-only for `t_rxd` in `NegotiatedTerms.__post_init__` so the `swap_state.py:265` ordering guard always fires, and have the gates decode the Radiant window via the same `nSequence` bit-22 encoding consensus uses rather than dividing by `block_interval_s`.

#### H2 — `(F-003)` R1 REF-authenticity gate is 100% delegated to one indexer; binding (b) is dead code, bindings (a)/(d) are an echo tautology, and there is no SPV/Merkle cross-check

- **Mechanism.** Radiant consensus only requires output singleton-refs ⊆ input outpoints (`validation.h:1044-1064`), with no mint/uniqueness/gly-envelope/value-conservation check — a self-crafted bare singleton is consensus-indistinguishable from a genuine Glyph. The sole off-chain defense is `verify_ref_authenticity`. Three structural defects: (1) `RxinDexerRefAdapter` hardcodes `has_gly_marker=True` for any non-None dict (`radiant_leg.py:218`), so the marker check at `ref_authenticity.py:190` can never fire in production; (2) the indexer receives the queried ref and re-encodes its own reported outpoint into `genesis_outpoint` (`radiant_leg.py:207/213`), controlling both sides of the equality at `:183`; (3) no SPV/header/Merkle cross-check exists; confirmations come from the same `chain_io`. The envelope-decoding RPC `glyph_get_metadata` exists (`rxindexer.py:133`) but is never called by the gate.
- **Evidence.** **PROVEN** by PoC against the real `verify_ref_authenticity`: a `LyingIndexer` echoing the queried ref as `genesis_outpoint`, `has_gly_marker=True`, the expected payload, and confs=100 PASSES the gate for a fabricated 36-byte REF for both `nft` and `ft` (with payload binding); the honest `None` control fails closed. The module docstring itself (`ref_authenticity.py:124-127`) concedes "A lying or attacker-controlled indexer defeats this gate."
- **Exploitability.** FT/NFT swaps only (the `rxd` variant short-circuits at `:143-147`; the 2026-05-27 dust runs were plain RXD with `indexer=None`, correctly unaffected). Requires a lying/compromised/MITM'd indexer (no cert pinning, no response signing) **or** an honest-but-buggy indexer that indexed a bare-singleton forgery — binding (b) being dead means even an honest indexer is not sufficient. The gate *is* wired un-skippably and fails closed on a single honest indexer's error/None, which is why this is HIGH not CRITICAL. **No FT/NFT real value has moved yet** — this is the value-moving path for the *next* FT/NFT swap.
- **Verifier verdicts.** All three lenses **CONFIRMED**.
- **Fix.** Before any above-dust FT/NFT swap: make binding (b) a real check (decode the gly envelope from the genesis reveal; never derive the marker from "the indexer returned a dict"); add an independent on-chain proof (fetch the genesis reveal + a header-anchored SPV/Merkle proof and verify the envelope + outpoint locally — the `spv/*` primitives already exist) and/or a quorum of independent indexers. Until then, gate FT/NFT mainnet swaps behind `require_audit_cleared`. Downgrade the docstring claim that the named bindings "reject a forged asset" to "reject an honest indexer's typo, NOT a dishonest indexer."

#### H3 — `(F-004)` Reorg-gate depth is anchored on server-supplied claim bytes; `get_raw_tx` never binds returned bytes to the requested txid (substitute-txid depth over-report) — **UNCERTAIN severity**

- **Mechanism.** The harness derives the trusted claim txid locally (`btc_txid_from_raw(broadcaster.last_raw)`, `dust_swap_run.py:323`) but then re-fetches the claim bytes via `get_raw_tx(btc_claim_txid)` and passes the **server-returned** bytes into `taker_scrape_and_claim_asset` (`:347/:354`), discarding the trusted txid. No `get_raw_tx` implementation asserts `btc_txid_from_raw(raw) == requested_txid` (`bitcoin.py:242-277` and the Blockstream/BitcoinCore/MultiSource variants). `confirmations_of_claim` then re-derives the txid from those *same* server bytes (`htlc_leg.py:396`) and reads that txid's confs. An attacker controlling/MITM'ing the endpoint, after `p` is public, can serve a same-preimage substitute tx with a different txid and over-report its depth.
- **Evidence.** **PROVEN** — substitute-tx constructed with the project's real `btc_txid_from_raw`/`scrape_secret`: real and substitute txids differ, `scrape_secret` recovers `p` from both, and `confirmations_of_claim(substitute)` returns the attacker-reported depth while the honest real-txid depth is 0. The txid = `hash256(non-witness serialization)` (`taproot.py:911-924`), so any non-witness mutation changes the txid while `p` stays scrapable.
- **Verifier verdicts.** exploit-construct + existing-guard: **CONFIRMED at HIGH** (real, breaks the reorg-gate guarantee in a way the documented SPOF mitigation does not catch). consensus-source: **CONFIRMED mechanism, severity HIGH→MEDIUM** — in the *as-wired single-source* dust path, the same endpoint can already over-report the **real** txid's depth directly (the documented and accepted SPOF, `bitcoin.py:809-814`), so the substitute adds no marginal power *today*; the distinguishing power ("defeats independent corroboration") materializes only once the project's own documented above-dust multi-source quorum confs reader is built. **Net: real defense-in-depth bug, latent escalation to HIGH the moment a quorum reader lands.** We carry it as HIGH with this caveat: it must be fixed *before* the quorum reader, or the quorum gives false assurance.
- **Fix.** Bind returned bytes to the requested txid at the trust boundary: in **every** `get_raw_tx` (including `MultiSourceBtcDataSource`), assert `btc_txid_from_raw(raw) == str(requested_txid)` (fail-closed `NetworkError`). Anchor the reorg gate on the trusted local txid (`broadcaster.last_raw`); use served bytes only for `scrape_secret`, with a matching-txid assert.

#### H4 — `(F-005)` BTC confirmation depth is a single mempool.space read with no internal consistency bound, consumed exactly once before an irreversible asset claim

- **Mechanism.** `MempoolSpaceFundingReader.confirmations` computes `confs = tip - block_height + 1` with both `tip` and `block_height` from the *same* endpoint, no `block_height <= tip` assertion, no PoW/header check, no second source (`bitcoin.py:944-951`). The coordinator reads this once (`swap_coordinator.py:897`) and on SAFE immediately fires `radiant_leg.claim_asset` (`:924`) with no re-read.
- **Evidence.** **PROVEN** — executed `assess_claim_finality`: honest `confs=1` → WAIT; the same shallow tx with a lying `confs=6` → SAFE (flip confirmed). Arithmetic probe: an over-reported tip (`+5`) inflates depth and flips WAIT→SAFE; the inverted `block_height > tip` case clamps to 0, so the proposed `block_height <= tip` assert alone does **not** catch the live over-report vector. `MultiSourceBtcDataSource` does not implement the `BtcFundingReader` protocol (`confirmations`/`read_output_amount_sats`/`txid_of`) and is not wired in; the harness wires the single-source reader directly.
- **Exploitability.** Not maker-triggerable; requires a network-level adversary (endpoint compromise, mis-issued cert, DNS/BGP redirect — TLS is enforced, `bitcoin.py:822`) or a genuine deep reorg the API hasn't reflected. The module note (`bitcoin.py:809-814`) explicitly accepts this as a dust-only SPOF and states above-dust value "MUST be backed by a multi-source quorum" — not built.
- **Verifier verdicts.** All three lenses **CONFIRMED**; severity HIGH on any above-dust run, with practical likelihood MEDIUM-HIGH today (elevated precondition + accepted-for-dust SPOF). Distinct from H3 (which substitutes bytes); H4 is the simpler single-transient-lie with no internal sanity bound.
- **Fix.** Back `confirmations_of_claim` with a quorum reader (adapt `MultiSourceBtcDataSource` to `BtcFundingReader`) and/or verify burial against independently-fetched block headers (SPV cumulative-PoW). At minimum assert `block_height <= tip`, re-read depth immediately before `claim_asset` and refuse on any decrease. Do not run above-dust value on the single-endpoint reader.

#### H5 — `(F-006)` SQUEEZED strands the asset: the harness loop treats ASSET_VULNERABLE as catch-all WAIT, re-calls a SECRET_REVEALED-only method, and crashes — the documented recovery and the #1 maker-stall defense are never invoked

The winner-take-all recovery `taker_claim_asset_from_vulnerable` (`:930`) and the proactive `maybe_refund_asset_on_maker_stall` (`:952`) are never wired into either harness (grep: 0 calls; the only `maybe_refund…` hit is a warning print at `dust_swap_run.py:313`).
- **Evidence.** **PROVEN** dynamically against the real coordinator + FSM fakes: call 1 returns ASSET_VULNERABLE (loop does not break); call 2 (the harness re-entry) raises the `ValidationError`; the recovery method `taker_claim_asset_from_vulnerable` from ASSET_VULNERABLE → COMPLETED (the call the harness fails to make). - **Exploitability.** In the single-operator dust model the operator plays both parties, so realized loss is self-inflicted and dust-bounded — the only reason this is not CRITICAL. Wire `maybe_refund_asset_on_maker_stall` into a background `asyncio` monitor that polls inside `maker_stall_safety_window_blocks` during BOTH_LOCKED. Add an integration test driving the coordinator to SQUEEZED and asserting a claim attempt (or clean handled exit), not an unhandled `ValidationError`.

#### H6 — `(F-008)` Resume inflates `asset_locked_at_height` to the current RXD tip (not the covenant's actual fund height), neutralizing the reorg-finality gate on every crash recovery

- **Mechanism.** The Radiant CSV refund opens at `actual_fund_confirmation_height + t_rxd`. The true fund height *is* available — `find_covenant_utxo` returns `(outpoint, value, height)` (`radiant_leg.py:164`) — but `covenant_outpoint`/`_resolve_covenant` (`:384/:397`) drop it, and no fund height is persisted in the keys file (`dust_swap_run.py:166-191`).
- **Evidence.** **PROVEN** numerically against the real `assess_claim_finality` (`t_rxd=20`, `btc_depth=2`, `rxd_burial=2`, true fund height 1000, refund opens 1020): at `now=1019` with a deep BTC claim, forward semantics (`asset_locked_at_height=1000`) → SQUEEZED (correct; blocks_left=1<burial); resume semantics (`asset_locked_at_height=1018`) → SAFE → silent automatic claim into a 1-block window the maker can race. **CONFIRMED-BY-SOURCE:** `d9c2224` fixed the analogous `t_btc`-drift-on-resume but missed this height drift; `dust_swap_resume` has zero test coverage. RESUME-2 corollary: the WAIT-loop deadline `0.5*t_rxd` from resume-start can extend past the real refund opening.
- **Exploitability.** Reachable on the real mainnet recovery path (resume exists precisely for the crash-while-BOTH_LOCKED case that produces an inflated tip). Tempered to HIGH (not CRITICAL) because it is an ops runner (not the shipped package), bites only with a late resume + reorg-shallow/adversarial maker, and current usage is single-operator. But it silently neutralizes the security-HIGH D4 control on a real-value path.
- **Verifier verdicts.** All three lenses **CONFIRMED** (consensus-source confirmed the BIP68 model: `CalculateSequenceLocks` opens the refund at `fund_confirmation_height + csv`, `tx_verify.cpp:103-129`).
- **Fix.** Derive `asset_locked_at_height` from the covenant UTXO's actual confirmation depth: surface the `_height` `find_covenant_utxo` already returns through `covenant_outpoint`/`post_asset_lock_revalidate`, and on resume set `asset_locked_at_height = now_rxd - covenant_confs + 1`. Cheaper alternative: persist the funded confirmation height in the keys file at forward-run time and refuse pre-fixup files (mirroring the `t_btc_blocks` fix). Bound the RESUME-2 deadline by `(refund_opens_at - now)`. Add a resume integration test that funds N blocks before resuming.

### UNCERTAIN

#### U1 — `(F-007)` Reorg-gate WAIT branch conflates BTC blocks with RXD blocks 1:1 — confirmed unit defect, refuted impact

- **Status.** The **code-level unit conflation is real and CONFIRMED**; the claimed MEDIUM stranding impact is **REFUTED**. Carried as UNCERTAIN → effectively LOW.
- **Mechanism.** `assess_claim_finality` normalizes with one `policy.block_interval_s` (the measured BTC median ~600s). `btc_depth_rxd = btc_claim_reorg_depth.normalize_to(BLOCKS)` is a no-op for a BLOCKS input, so the WAIT test `blocks_left - btc_depth_rxd >= rxd_burial` (`:531`, confirmed above) subtracts a count of BTC blocks (~600s each) from a budget of RXD blocks (~300s each), under-debiting the RXD window by the BTC/RXD ratio. The plan specified "N_btc (btc→rxd-converted)" and flagged this; it was not implemented. No test uses distinct intervals.
- **Why the impact is refuted.** **PROVEN** by simulation: (1) **no false SAFE is possible** — SAFE is a separate branch entered only when the BTC claim is already reorg-deep, and its only test (`blocks_left >= rxd_burial`, `:525`) is correctly RXD-vs-RXD with no `btc_depth` subtraction; the conflation lives solely in the WAIT/SQUEEZED split, and SAFE is the only branch that fires `claim_asset`. (2) **Self-corrects in one poll** — `now_rxd_height` is re-read live each loop, so a mislabeled WAIT flips to SQUEEZED the instant one RXD block is produced, with ample window remaining; ASSET_VULNERABLE is recoverable and the loop deadline is independently capped. No call/timing sequence strands or steals funds.
- **Fix (low priority, precision/defense-in-depth).** Carry `rxd_block_interval_s` onto `MarginPolicy`, convert the BTC reorg-depth wait into RXD blocks (`ceil(btc_blocks_remaining * btc_interval_s / rxd_interval_s)`, rounded up) before subtracting; add a test with distinct intervals.

#### U2 — `(R2-0-0-F-019)` `FeeInput` leaks the WIF in its default dataclass repr — confirmed leaky repr, no reachable sink today

- **Status.** The **leaky repr is real and CONFIRMED**; no exploitable path exists today and the marginal blast radius is ~zero, so severity MEDIUM → **LOW** (defense-in-depth hygiene). Carried as UNCERTAIN.
- **Mechanism.** `FeeInput` is a frozen dataclass with `wif: str` and no `field(repr=False)` (`htlc_spend.py:56-71`); the default `__repr__` renders the WIF. The mitigation pattern exists in-repo (`hd/wallet.py:228-229` uses `field(repr=False)`; `security/secrets.py:90-91` has `SecretBytes`) but was not applied here. On the mainnet harness the WIF is real wallet key material from `dumpprivkey` (`radiant_mainnet_chainio.py:170/206`).
- **Why it is not exploitable today.** **PROVEN** — reproduced the repr leak in isolation, but a grep across `src/` and `scripts/` found **no** `logger`/`print`/`%s`/`!r`/`asdict`/`json.dump` path that stringifies a `FeeInput` or `fee.wif`; no `faulthandler`/`sys.excepthook`/locals-dumping config. The same wallet WIF is already written cleartext in the mode-600 keys file alongside every other key and the preimage (`dust_swap_run.py:166-191`, self-described "single point of total compromise"), so any observer who could win the hypothetical repr leak can already read that file.
- **Fix.** `wif: str = field(repr=False)` (mirror `hd/wallet.py:228`), or wrap in a `SecretStr`-style holder; add a `'wif' not in repr(...)` unit test. Cheap; do it regardless.

## REFUTED Findings (valuable negative signal)

These were investigated and found **not exploitable** as stated. Recording them prevents re-litigation.

- **`(F-009)` Maker fee-bumps a CSV refund to win the mempool race for the covenant UTXO — REFUTED.** All static facts hold (SAFE at `blocks_left >= rxd_burial` with no separate inclusion-slack term; COMPLETED-on-broadcast with no burial re-check; non-RBF claim sequence; signatureless refund). But the headline mechanism is impossible on Radiant: **Radiant has no RBF** — `AcceptToMemoryPool` returns `REJECT_DUPLICATE "txn-mempool-conflict"` for any input conflict regardless of fee and builds a DSProof (`validation.cpp:610-668`); it is first-seen. Once the taker's claim is in the mempool the maker's refund is rejected on submission. The refund's BIP68 lock is also consensus-enforced, so it cannot enter a block before `refund_opens_at`, giving the taker a `rxd_burial(≥2)`-block window as the *sole* valid spender. The GRIEF-5 "BIP68 off-by-one" sub-claim points the **wrong way** (the gate is already conservative by one block). **Residual genuine gap (→ defense-in-depth, not the claimed HIGH):** zero inclusion-slack + terminal COMPLETED-on-broadcast means that *if* the taker's flat-fee claim is **evicted** from all mempools (node restart without rebroadcast, or fee-floor eviction under extreme congestion) for the full burial window, the matured refund could be mined while the record falsely reads COMPLETED. That is an eviction/liveness contingency on taker-node failure, not maker-fee theft. Worthwhile hardening (post-broadcast burial gate before COMPLETED; positive inclusion slack; RBF/CPFP bump path), but the RBF and `-1` items rest on incorrect premises for Radiant.
- **`(F-012)` Forged mempool.space headers shrink the cross-chain margin to the 1-block floor — REFUTED.** Mechanically true (headers are unauthenticated, no PoW/chaining/second source; a forged near-uniform timestamp series collapses `margin_blocks` to 1). But: **harm direction is inverted** — a smaller margin makes `t_btc` *smaller*, opening the taker's BTC refund *sooner* and *shrinking* the maker-stall steal window (8→5 blocks, sim-confirmed). The runtime claim-safety gate `assess_claim_finality` reads only `t_rxd` + operator-set reorg depths, **never `policy.margin`**, so a forged margin cannot weaken it. `margin_blocks=max(1,…)` + hardcoded `+4` slack guarantees `t_btc >= t_rxd + 5` for *any* forged series. The interval pollution is inert for BLOCKS→BLOCKS normalization. In the single-operator harness the header-fetcher is the protected party (self-harm). Header authentication / a margin sanity floor are reasonable hardening but not an exploitable issue → INFORMATIONAL.
- **`(R2-0-2-F-015)` Deprecated SPV-oracle path has no R1 gate and is importable — REFUTED as a vuln.** Literally true (gate called only from `SwapCoordinator`; `trade.py`/`maker.py`/`receive.py` run no R1; all importable). But the R1 forged-singleton attack **requires a ref-bearing covenant**, and the SPV-oracle path structurally cannot carry one: `GravityOffer` has no genesis_ref/singleton field (locks plain RXD photons); a push-aware opcode scan of every `maker_*` artifact yields zero real ref opcodes (0xd0–0xe0); ref opcodes exist only in the HTLC artifacts. There is no REF to authenticate, and the gate is a no-op for the no-ref variant anyway. Code hygiene (a hypothetical future ref-bearing covenant on `GravityTrade` would miss the gate), not a present hole → INFO.
- **`(R2-0-3-F-016)` Value-path BTC parsers have no fuzz coverage — REFUTED as a vuln (coverage gap is real).** The hypothesized harm (a parse divergence yielding a wrong-but-deep txid) is structurally unreachable: `btc_txid_from_raw` re-serializes verbatim (`taproot.py:957-996`), so for any node-accepted tx the Python txid equals the node txid by construction; the only inputs Python accepts but a node rejects are non-canonical CompactSize encodings, which Radiant's `ReadCompactSize` throws on (`serialize.h:419-444`) → such a tx never confirms → the gate reads 0 confs (fail-closed); a wrong-but-collision-free txid needs a `hash256` second-preimage. Property/differential tests are cheap hardening worth adding, but the MEDIUM (silently broken reorg gate) is unjustified → INFO/test-hygiene.
- **`(R2-2-1-TOCTOU-2)` SeenStore documented durable but in-memory and never invoked on resume — REFUTED as exploitable.** All structural facts true (and overlap H7's documentation gap). But no exploit can be constructed: atomic-swap safety is independent of the seen-store — the maker can claim BTC only by revealing `p` on-chain, after which the taker scrapes `p` by hash-match and claims the matching covenant; refund safety is the consensus-enforced CSV/timeout, not the set; reusing one `H` across two swaps hands the taker the keys to claim both RXD legs from one reveal (no monetizable asymmetry); the free option is bounded per-leg by the independent timelock. The "never invoked on resume" fact is harmless — resume begins at BTC_LOCKED (funding already on-chain); a pre-funding admission check has nothing to protect post-commit. Doc-hygiene/hardening (fix the docstring or implement durable + wire), not an exploitable vuln. *(Note: H7 stands as the* economic *free-option / cross-swap-replay angle on the shipped two-party coordinator, which this refutation does not cover.)*
